:lock: feat(bridge): v0.0.7 — Skill as hard dependency, alive-file, /toraseo command

Closes the v0.0.7 Bridge Mode MVP by enforcing three product
invariants discovered during dogfooding:

1. The Skill is a HARD dependency. Without it, Bridge Mode
   cannot complete. The protocol token lives only in SKILL.md
   §2.1 and the MCP server's constants — never in the prompt
   the App generates. Previously the prompt embedded the token
   and any model with MCP access could pass handshake by reading
   chat (Haiku reproduced this without the Skill installed).

2. App-running detection is independent of scan state. A new
   alive-file (%APPDATA%/.../app-alive.json) carries PID +
   heartbeat so MCP can distinguish three coarse states:
   `app_not_running`, `app_running_no_scan`, `in_progress`.
   Previously MCP could only check current-scan.json, which
   meant "no scan started" looked the same as "app not running".

3. Bridge Mode triggers are explicit, not heuristic. A new
   /toraseo (and /toraseo bridge-mode) slash-command is the
   primary trigger, with transliterated mentions (ТораСЕО,
   ТораСэо, Тора СЕО, Тора Сэо, etc.) as secondary triggers.
   The auto-prompt copied to clipboard now starts with
   "/toraseo bridge-mode" for unambiguous detection.

Files added:

- app/electron/bridge/aliveFile.ts
  setupAliveFile / teardownAliveFile / detectExistingInstance.
  Creates app-alive.json on app ready, writes heartbeat every 10s,
  removes file on before-quit. Uses atomic write (tmp + rename)
  same as state-file. detectExistingInstance probes for a
  pre-existing file from a previous run; if PID is dead, treats
  as stale (free to overwrite); if alive, logs a warning and
  continues (single-instance enforcement deferred).

- mcp/src/aliveFile.ts
  probeAppAlive() reads the alive-file and returns a discriminated
  union: { kind: "running", pid, version, startedAt } or
  { kind: "not_running", reason: "no_file" | "stale_pid" |
  "stale_heartbeat" | "read_error" }. PID check uses
  process.kill(pid, 0) (no-op probe). Heartbeat freshness threshold
  is 30s (= 3x the 10s heartbeat interval). Same dual-candidate
  path discovery as stateFile.ts.

- mcp/src/verifySkillLoaded.ts
  Three-stage diagnosis instead of binary success/no_scan:
  - Try handshake against state-file.
  - On no_scan, probe alive-file → app_not_running (with reason)
    or app_running_no_scan or wrong_state.
  - On mismatch, return token_mismatch.
  - On verified, return scan params.
  Each error response carries a code and a message instructing
  Claude how to react. The app_running_no_scan response
  explicitly tells Claude to use ask_user_input_v0 with two
  options (chat fallback / wait for Scan) and to NOT call any
  tools without explicit choice — this implements the user's
  product requirement that Bridge Mode never silently runs.

What this does NOT solve (deferred):

- Single-instance enforcement on app start. detectExistingInstance
  reports kind:"alive" but startup proceeds anyway. A future
  release can call app.quit() with a user-facing dialog.
- Heartbeat may give false negatives on systems where Node's
  setInterval is suspended (laptop lid close, OS sleep). Falling
  back to PID-only check would be safer; for now, the 30s threshold
  is forgiving enough that brief suspensions don't trigger it.
- mcp_status() tool returning {app_running, skill_compatible,
  protocol_version} — would give Claude richer diagnostics but
  doesn't change the reactive nature of MCP. Tracked for v0.0.8+.
- Slash command /toraseo conflict detection. Future versions may
  add /toraseo content, /toraseo image, etc. — for now there's
  only one slash-command and no conflict-checking infrastructure.
