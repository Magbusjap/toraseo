:wrench: fix(ci): split release tags into app (v*) and skill (skill-v*) namespaces

The previous setup tried to use a `-app` suffix on tags
(e.g. v0.0.2-app) to differentiate app releases from skill releases.
This produced an orphan release: electron-builder ignores the trigger
tag and creates its own release named `v{version}` from package.json,
so a tag like `v0.0.2-app` resulted in two releases — one empty
(`v0.0.2-app` from our workflow) and one with the actual installer
(`v0.0.2` from electron-builder).

Fix: clean separation of release tracks via tag namespaces.

- App track uses plain version tags (v0.0.3, v0.1.0, v1.0.0).
  electron-builder is happy because the tag matches the version it
  computes from package.json.
- Skill track uses the `skill-v*` namespace (skill-v0.2.0,
  skill-v0.3.0). softprops/action-gh-release strips the `skill-`
  prefix when building the asset filename so the ZIP keeps reading
  `toraseo-skill-v0.2.0.zip`.

Workflow changes:
- release-app.yml: trigger `v*` with `!skill-v*` exclusion
- release-skill.yml: trigger `skill-v*` only; computes asset name
  by stripping the prefix; comment block explains the split

CHANGELOG additions:
- New section "Two release tracks" documenting the split for any
  future contributor or reader
- New entry [App 0.0.2] describing what shipped (auto-update
  infrastructure: electron-updater, electron-log, in-app
  notification, IPC contract, CI workflow)
- [Unreleased] updated with the next slice of work — hard-dependency
  detector + onboarding screen for app v0.0.3+

Wiki updated with the same correction in
auto-update-implementation.md including a record of the original
mistake so we don't reproduce it.

Legacy v0.1.0-alpha (skill) keeps its plain v* tag — won't move it
post-release.
